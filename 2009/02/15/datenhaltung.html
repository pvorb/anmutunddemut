Datenhaltung

In letzter Zeit beschäftige ich mich – eigentlich ohne große Not – mit unterschiedlichen Konzepten der Datenhaltung. Ein bisschen was davon möchte ich hier mal zusammenfassen. Erstmal nur so für mich. Ohne Sendungsbewußtsein und Anspruch auf Vollständigkeit oder Korrektheit.

Relationale (SQL basierte) Datenbanken

Relationale Datenbanken sind immer noch in gewisser Hinsicht das A und O der Datenhaltung. Klassischer Weise überlegt sich der Entwickler dabei vorher welche Informationsstrukturen man denn gerne abbilden möchte und bastelt dann daraus mit viel Gehirnschmalz und evtl. einem Tool wie dem WWW-SQL-Designer ein normalisierte Datenbankschema. Darauf baut dann die Anwendung auf. Soweit so gut.

Vorteile

Es ist hocheffektiv und prima performant
Es ist gut filter- und sortierbar.

Nachteile

Google hat keinen direkten Zugang zu Datenbanken.
Datenstrukturen können nicht direkt von Laien erstellt oder erweitert werden.

Content Construction Kit kurz CCK

CCK ist eigentlich ein Drupal Modul, dass man sich so aber auch für jedes gute CMS bauen könnte. Es erweitert das oben beschriebene Model relationaler Datenbanken. Die Idee ist ganz einfach: Man kann über ein GUI zu einem bestehenden Content-Typen (Wie bspw. Blogartikel, Tweets, Bookmarks, etc.) weitere Datenfelder hinzufügen. Und zwar alles was das Herz begehert: Datumse, Zahlen, kruze Texte, lange Texte, URLs, Dateienm Flickr-Bilder, Tweets – kurz: Alles. CCK legt dazu dann einfach weitere Datenbanktabellen an und fertig ist die Laube.

CCK wandert in Drupal 7 nicht umsonst in den Kern, denn es erfüllt einen Bedarf, der bei immer mehr Betreibern von Websites aufkommt: Man möchte nicht nur Artikel schreiben sondern den Kram über den man da schreibt auch sauber erfassen und hinterher sauber weiterverwenden. Ich glaube CCK wird als generisches Feature eine große Zukunft haben, und andere CMSe werden es schnell adaptieren. Uuund: CKK und das Semantic Web sind dufte Kumples.

Nachteil

Es ist eher inperformant.

Vorteil

Es kann von Laien erstellt und gepflegt werden.

XML

XML ist eigentlich zu abstrakt um direkt mit relationalen Datenbanken vergleichbar zu sein. In seinem konkreten Einsatz ist es aber in einigen Punkten durchaus vergleichbar. Wie HTML hat XML seine Wurzeln in einer dokumentzentrierten Vorstellung von Daten. Und es spielt immer dann sein Stärken voll aus, wenn es zur Strukturierung von Dokumenten verwendet wir.

Nachteile

Volltextsuche / Aspekt-Suche über ein ganzes Korpus geht nicht wirklich gut.
Für Korpora ist es noch eher inperformant.

Vorteile

XML-Dokumente sind als URLs erreichbar.
Erlaubt es Mixed Content Modell abzubilden (OHCO-Modell).
Von XML-erfahrenen User sehr leicht auf Dokumentebene erweiterbar.

DB-XML

Für das kleine Itunes-Experiment von neulich hab ich auf dem XML-Export-Format von Itunes gearbeitet. Ich hab keine Ahnung, ob Itunes das intern auch verwendet. So wie Itunes sich verhält scheint es mir aber fast so. Das ist durchaus erstaunlich und versetzt mich seither ins Grübeln, denn dieses XML-Format ist in einem sehr hohen Maße generisch. Man könnte es in einem relationalen DB-Schema mit zwei bis vier Tabellen totschlagen.
Das erstaunliche daran, ist, dass Itunes alle komplexeren Operationen – genau wie ich in meinem kleinen Skript – auf mehrdimensionalen, assoziativen Array zur Laufzeit zu machen scheint. Klar, eine Desktop-Anwendung kann sich das erlauben. Aber mein kleines PHP-Skript scheint damit auch keine Probleme zu haben.

Nachteile

Das Modell beinhaltet tonnenweise Redundanzen

Vorteile

Es ist extrem leicht erweiterbar und ähnelt dem Ansatz von CCK

Nachtrag: Der Artikel lag jetzt gut 14 Tage als Draft rum und ich hab ihn immer wieder hier und da erweitert. Und je länger ich ihn hab abhängen lassen, desto mehr wuchs in mir der Wunsch, doch wieder ein eigenes CMS hochzuziehen. XML ist so geil und Datenbanken so unkuhl. Außerdem will ich wieder alles selber kontrollieren. Alles. Und wenig Code haben. Minimalismus muss.

Mit „DB-XML“ meinst Du die ~/Music/iTunes/iTunes Music Library.xml? Wenn ja, dann ist das nur eine Metasyntax für hierarchische Datenstrukturen, ähnlich wie JSON oder das schönere YAML. In diesem Falle handelt es sich um property lists, einer Schreibweise für fast die gleichen Datenstrukturen (Listen, Dictionaries, Zahlen, Strings, Wahrheitswerte), die es auch in JSON gibt. PLists wurden für NextStep als ASCII-Struktur mit geschweiften Klammern eingeführt, mit der Übernahme in Mac OS X dann ver-XML-t und inzwischen gibt es eine Binärversion davon um schneller darauf zugreifen zu können. Für PLists gibt es in Apples Frameworks haufenweise Parser, die daraus richtige Objekte machen; deswegen wird das dort gerne zur Datenspeicherung genommen. Rein auf der syntaktischen Ebene, als potentiell Schreibender ziehe ich bei so etwas YAML vor.

Weg von der Syntax, hin zu Datenbanken, die Dich eventuell auch interessieren könnten, nämlich zwei Sorten von Dokument-orientierten Datenbanken. Man arbeitet bei diesen nicht zwangsläufig mit einem strengen Schema; kann also nach belieben weitere Daten hinzufügen.

XML-Datenbanken wie exist speichern XML-Dokumente, bieten zusätzlich auch noch einen Index über die XML-Struktur an. Man kann dann mit XPath oder XQuery wunderbar Anfragen an die Menge von Dokumenten stellen und diese als neue XML-Dokumente ausgeben. Manche XML-Datenbanken bieten auch integrierte Volltextsuchen an.

Eine andere, derzeit ziemlich hippe Dokument-orientierte Datenbank ist CouchDB; in dieser werden versionierte JSON-Dokumente gespeichert. Spezielle Views auf diese Dokumente werden mittels serverseitigem Javascript erstellt, das dann natürlich direkt auf JSON operieren kann. Auch eine Volltextsuche ist in Arbeit.

Sowohl exist als auch CouchDB bieten REST-Interfaces über HTTP an. Dazu kommen recht mächtige Admin-Interfaces, die wiederrum auch über HTTP funktionieren. Natürlich setzt dass dann einen eigenen (v)Server voraus; auch sind sie in einigen Teilen noch eher komplex, eher was für Bastler und ähnlich veranlagte; weniger was für die CMS-gewöhnten. Aber sie sind interessant.

Und natürlich gibt es Triple-Stores, die direkt RDF speichern. ;-)

(Kein <abbr>, <dfn>, <var>, <blockquote>?)

Text-Plists waren ja eigentlich mal eine Metasyntax, die Einführung von XML-Plists hat man vermutlich aus der Vermutung gemacht, dass eine eigene Syntax nicht so zugänglich ist, wie eine allgemein bekannte Metasyntax wie XML. Und so hat man ein Metaformat für Datenstrukturen in einer Metasyntax implementiert und wirft damit alles aus dem Fenster, was XML praktisch macht. ;)

Wenn Du auf der Suche nach simpel bist, ist vielleicht noch dieser Gedanke interessant: Dass das Dateisystem die simpelste und eine der schnellsten Datenbanken überhaupt ist. Es gibt einen Haufen kleiner „CMSe“, die statische Seiten, inklusive Feeds und ähnlichem generieren und auf den Server frachten. Im wesentlichen Skripts, die spezielle Text-Dateien parsen und in HTML und XML wandeln. Brent Simmons hat im Januar in seinem Weblog ein eigenes System vorgestellt und diverse andere verlinkt. Vielleicht sind da Anregungen drunter.

Noch ein Beispiel: Github hat Github Pages eine Möglichkeit, HTML zu publizieren, indem man einfach mittels git seine Änderungen im Repository auf Github hochlädt und somit als Nebeneffekt die dazugehörigen Webseiten publiziert. CMStechnisch können die Änderungen dann durch wieder so einen Generator statischer Seiten, nämlich Jekyllgejagt.

Ich finde an solchen minimalistischen Ideen toll, dass man sein Zeug dort bearbeitet, wo man sich am wohlsten fühlt, für mich dann im eigenen Texteditor in einem minimalsten Metaformat wie z.B. Markdown oder Textile oder Restructured Text, oder, oder, oder, sprich in Textformaten, die der Eingabe angemessen sind und sich nicht mit Textfeldern in Browsern abmühen muss. Das ist aber auch eher eine persönliche Vorliebe, andere mögen anderes lieber.

Ein weiteres Problem bei statischem ist natürlich der Mangel an dynamischen; keine Kommentare, keine Suche. Aber im Prinzip sollen all diese mini-CMS-Skripts auch nur Anregungen für ein mögliches Backend Deines Minimalst-CMS sein.
