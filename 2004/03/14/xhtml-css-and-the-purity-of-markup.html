XHTML CSS and the purity of Markup

During the past Months I had to polish my knowledge and abilities in HTML. As I am at first place working with XML-technoliges and only at the second place familiar wih HTML and CSS for a few years now, I was quite happy, the webdesigners all around the web, are more and mor interested in making their websites valid to some documentgrammar, and what's more important to keep it free from tagabuse.

But, what's quite funny, is the fact, that there are some problems with keeping XHTML valid, free of tagabuse AND stay true to the concept of "semantic markup":

HTML was designed to markup linear scientific text, that could form a hypertext. And if you've ever seen a scientific article, you shure know, that it's not [typo]graphically challanging.
So HTML was desing to markup Texts, which presentations were meant to be very close to printtext. As HTML had a rocket-like carreer not only being the main Language of the WWW but also coming into the computer-age of multimedia and highly-sophisticated Interfaces, HTML quickly had to do much more than only structure and present scientifc articles:

Webdesigners and User developed Intros, Menus, Navigations, Orientations, complex Informationarchitectures for huge Websites, gamelike plugins; in a few words:

An almost whole new 2D [and sometimes even 3D] medium.

Of course HTML is not fully appropriate to markup all these functions. Actually it should not even be almost appropriate to do so. But, hey, considering, it once was developed to markup tyoprgaphically and structurally boring [most times even textually boring] scientific articles, it has done quite well.

HTML along with XML has general problems in marking up
things like websites:

HTML and XML are both desinged to markup text.
Mostly linear text. Therfore it's main structur is tree-like, include the "depth-before-width" mechanism to get it linear.Webdesign, Interfacedesign and Informationdesign, are at least twodimensional, if not threedimensional or [incase of Information Architecture] n-dimensional. If you try to press structures like these in a tree, you'll hav a hard time.

Menus usually are o the left or on the right of the ?main? content, but in HTML you HAVE to place them befor or afterwards the content. But there a no good reasons for choosing neither one or the other way. But on the otherhand the order of the ?p"-elements inside your content is important, so somewhere is a gap in the semantics of your content?

This problem is even more obvious at the places where XML tries to markup somethings that's completly un-tree-like: like topic Maps. Sure, you can mark it up, but you have a hard time, getting the information out there in a meaningfull way...

The set of Elements and Attributes, that come up with HTML is nolonger appropriate to the different semantic objects, that are contained inside a website. So webdesigners
have to look for workarounds: They use blind-tables or divtags, or spans, or whatever...

Now the followers of the path of the pure Markup say, "this is bad! You didn't completly divide Markup from presentation."

And I say: Sure, I can't. Theres now way to design a state-of-the art website, without leaving the text-orineted-paradigm of HTML. So in a way, you have to hack HTML. But that's no problem, as long as you do not leave the unspoken agreement between the W3C, webdesigners and HTML-Exploiters, like browsers-programmes or searchengine-programmers. This agreement is about the "meaning" of parts of your markup. "h1" should be used to markup the name of your page; "strong" should be used to markup inlinetext which seperates from it's surrounding text in a way that it is more important... *wow*...

For the W3Cs sake HTML has the "div" and the "span" Element to group and markup any of your data, if you need it. The only common-meaning, is, that it groups the content inside it's tags together... They could be used unviersally.

What's a real lack in HTML is the missing of futher
general Attributes besides "id" and "class" like "type", "meta", "topic", "index", which would really help dividing the different functional levels of a website. As long as we don't have a unviersal Markuplanguage of 2D/3D interactive Interfaces for nonlinear, multimedia Content, we have to workaround with HTML in the way, that we use general Elements and Attributes, so the supprot for that could be a bit better...

I?m not sure wether there will be a better semantic Markuplanuguage for the WWW than HTML, but right now it?s to early to design it, because our medium ist still to young, and things develop to fast, to design an allembracing Markup of 2D/3D interactive Interfaces for multimedia Content.

We have to workaround with HTML but that does not mean, that we ought to produce tag-soup all along. Use HTML, as semantic as it gets, and if you reach the borders of it, hack it! But keep semantics in mind.

If you act like this, you?ll get 99% XHTML 1.1. valid Markup, that semantically quite fine, but not maybe not as good as it might be in a far more advanced Markuplanguage. So I think,there?s no need for ?the myth of css? or any semi-religous battles around the purity of Markup?
